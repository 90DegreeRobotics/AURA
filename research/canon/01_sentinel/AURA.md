# AURA System Architecture

The AURA system architecture processes user inputs from the AURA UI, routes and processes them through the AURA RUNTIME, secures and validates them within the SENTINEL-CORE, and finally records the definitive outcome in the Immutable Audit Log and Canonical Truth Engine.

## 1. AURA UI (User Interface)
The top layer, AURA UI, is the user-facing component of the system, built as an Electron shell. It is composed of four main modules:

*   **Chat & Voice:** Enables direct interaction with the system via text chat or voice commands.
*   **Quiz & Preferences:** Allows users to set personal preferences, manage login, and participate in the Mirrorborn quiz flow.
*   **Notes & Calendar:** Provides tools for users to manage their personal notes, events, and calendar entries.
*   **Archetype Engine:** Suggests a component for defining or personalizing user profiles and interaction styles.

Commands and requests generated by these modules are sent down to the AURA RUNTIME.

## 2. AURA RUNTIME
The second layer is the AURA RUNTIME. It acts as the central processing unit for user inputs, managed by its "Policy Enforcement & Tool Router". It accepts requests from the UI, turns them into a canonical action request, calls Sentinel-core for “allow/deny/require-consent,” and only then performs the action. After execution, it writes the outcome back as another event. If Sentinel-core is down, actions fail closed and loudly.

This layer contains four key components:

*   **Command Processor:** Interprets and executes commands received from the UI.
*   **LLM Adapter:** Connects to a Large Language Model (LLM) for advanced language processing and generation.
*   **STT / TTS Module:** Handles Speech-to-Text (STT) and Text-to-Speech (TTS) conversion for voice interactions.
*   **Notes & Calendar Tools:** Specific tools that manage the backend logic for notes and calendar functions.

The AURA RUNTIME processes "Commands & Requests" from the UI and sends "Action Requests" to the Sentinel-Core. It also receives "Action Results" back from the core.

## 3. SENTINEL-CORE (Identity & Security Substrate)
The third layer is the SENTINEL-CORE, described as the "Identity & Security Substrate." It is the only privileged substrate, responsible for the security, identity management, and integrity of all system actions. It consists of four essential components:

*   **Identity & Nonce:** Manages user identities and uses nonces (a number used once) for secure authentication and replay protection.
*   **Policy Engine:** Enforces security policies and rules to determine if an action is permissible.
*   **Capability Manager:** Oversees the permissions and capabilities granted to different users or components.
*   **Append-Only Ledger:** A secure, immutable record that logs all transactions and events that everything must go through.

Based on its components, the SENTINEL-CORE makes decisions on "Action Requests" ("Allow / Deny / Consent") and logs all activities as "Audit Events".

## 4. Immutable Audit Log & Canonical Truth Engine
The final layer at the bottom represents the ultimate record of truth and accountability for the entire system. It receives outputs from the SENTINEL-CORE.

*   **Immutable Audit Log:** A permanent, unchangeable record of all "Audit Events," providing a verifiable history of every action taken within the system.
*   **Canonical Truth Engine:** This component works in conjunction with the audit log to establish and maintain the single, authoritative version of the truth for all system states and events.
